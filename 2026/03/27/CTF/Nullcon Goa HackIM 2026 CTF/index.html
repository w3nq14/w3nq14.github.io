预订密钥

赛事： Nullcon Goa HackIM 2026 CTF
类别： 加密货币
得分： 113
服务： 52.59.124.14:5102

概述

该服务返回一个整数列表，并要求输入对应的密码。挑战是一个基于已知文本（《爱丽丝梦游仙境》）的书籍密码。每个整数表示从当前光标位置到书中下一个密码字符出现位置的距离，并循环到书末。

关键见解

如果已知书的内容，则每个起始位置都定义了一条唯一的路径：

  • 对于距离 d ，下一个字符是 book[(pos + d) % L] 。
  • 距离必须与该字符从当前位置的下一个出现位置的距离相符。

这样就可以暴力破解起始索引并恢复完整密码。行尾符很重要，所以我们尝试了 LF 和 CRLF 两种版本的书籍。

解决方案

  1. 加载书籍文本（或提示），并为每个字母构建到下次出现位置的距离表。
  2. 对于每个可能的起始索引，遍历密码距离并检查一致性。
  3. 将生成的密码提交给服务器三次。
  4. 读取最终输出（标志）。

类别： 加密货币
得分： 50

概述

该服务会打印 reduce(flag, f) ，其中 f 是一个随机的 32 位多项式。这种归约恰好是一个 CRC 校验码，因此每个输出都给出了 flag 多项式模 f 的同余式。

关键见解

在 GF(2)[x] 中，每个样本给出 flag = r (mod f) 。有了足够多的互质随机模数，我们可以利用 GF(2)[x] 上的中国剩余定理将它们组合起来，从而恢复完整的多项式（即 flag）。

解决方案

  1. 打开多个连接并收集 (r, f) 对。
  2. 在 GF(2)[x] 上实现多项式运算和 CRT。
  3. 合并全等式，直到模数次数超过标志长度。
  4. 将多项式转换为字节并找到 ENO{...} 。

赛事： Nullcon Goa HackIM 2026 CTF
类别： 加密货币
得分： 50
服务： 52.59.124.14:5101

概述

服务器使用 Z_65 上的仿射变换对 base64 文本进行加密：

字母表是包含 65 个符号（包括 = ）的自定义 base64 集，纯文本填充为 16 个符号的块。

关键见解

加密预言机使我们能够恢复 b 和 A 的每一列：

  • 对 16 个 'a' （索引 0）进行加密，得到 b 。
  • 将单个位置替换为 'b' （索引 1）即可显示 A 的相应列。

因为 65 = 5 * 13 ，所以我们对 A 取模 5 和 13，然后与 CRT 重新组合。

解决方案

  1. 发送 17 个选定的明文块以恢复 b 和 A
  2. A 取模 5 和 13 的倒数。
  3. 对于每个密文块，分别用模数和余弦定理求解 A * x = c - b ，并与 CRT 结合。
  4. 重新构建 base64 字符串并对其进行解码。

赛事： Nullcon Goa HackIM 2026 CTF
类别： 加密货币
得分： 107
服务： 52.59.124.14:5104

概述

该服务使用 AES-CBC 加密标志，并使用 RSA (e=65537) 加密 AES 密钥。服务器充当阈值预言机：如果 RSA 解密后的密钥为 >= 2^128 则返回特定的错误消息；否则，继续执行 AES 加密并返回填充错误。

关键见解

RSA 是乘法算法：发送 c' = c * s^e mod n 得到解密后的 m' = m * s mod n 。预言机判断 m' < 2^128 。对于较小的 s ，这是一个单调条件，因此对 s 进行二分查找可以恢复精确的边界，从而得到 m 。

解决方案

  1. 解析 n 和来自服务器的密文。
  2. 使用 c' = c * s^e mod n 定义预言机。
  3. 二分查找，找到使预言机返回 true 的最大 s 。
  4. 计算 m = floor((2^128 - 1) / s_max) 。
  5. 使用 m 作为 AES 密钥并解密 flag。

要求

  • Python 3
  • cryptography

类别： 加密货币
得分： 251

文件： chall.py

概述

该服务实现了一种自定义的类 AES 分组密码，并公开了一个加密预言机。启动时，它会打印 encrypt(key, key) ，然后允许使用同一个 16 字节的密钥进行选择明文查询。该密码使用带碰撞的 S 盒，使其具有非注入性，并支持密钥恢复。

漏洞

S 盒不是排列组合：

  • S[0x00] == 0x64
  • S[0x8c] == 0x64

因此，在 SubBytes 步骤中，输入 0x00 和 0x8c 会发生冲突。通过构造输入，强制进入 SubBytes 的单个字节为 0x00 或 0x8c ，我们可以检测冲突并为每个密钥字节导出两个候选值。

关键复苏策略

  1. 对于每个字节位置 pos ，找到一个明文字节 p 使得：
    • encrypt(block with p at pos) == encrypt(block with (p ^ 0x8c) at pos)
  2. 这意味着该位置的 SubBytes 输入为 0x00 或 0x8c ，从而为相应的密钥字节提供了两个候选值。
  3. 对于字节 0，初始轮次也与 42 进行异或运算，因此候选结果为：
    • p ^ 42 和 p ^ 42 ^ 0x8c
  4. 总共有 2^16 密钥候选对象。在启动时使用泄露的 encrypt(key, key) 函数，通过本地重新计算来选择正确的密钥。

概述

加密器应用自定义的逐字节映射，与已处理的密钥字节进行异或运算，然后再次应用相同的映射。输出结果为 base64(processed) : sha1(processed) 。分支覆盖率揭示了执行了哪些密钥字节分支，因此密钥字节的无序集合被泄露。

关键观察结果

  • 密钥长度为 9 字节（ generateRandomKey(9) ）。
  • 覆盖率准确地识别出哪些 if ($keyChar == chr(n)) 分支被命中，从而得到 9 个原始键字节（无序）。
  • 按字节映射将每个原始密钥字节转换为 XOR 中使用的已处理密钥字节。
  • 同样的映射也适用于密文字节，因此每个字节都有一小部分原像。

解决方案

  1. 使用 sha1(base64_decode(data)) 验证完整性。
  2. 从覆盖率中提取 9 个原始密钥字节，并构建逐字节映射表。
  3. 将它们转换为 9 个已处理的关键字节。
  4. 计算每个密文字节的所有有效原像。
  5. 使用 ENO{...} 格式和可打印输出对已处理密钥字节的 9! 种排列进行暴力破解和过滤。
